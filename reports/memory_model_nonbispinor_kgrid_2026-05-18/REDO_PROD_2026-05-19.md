**Verdict.** Planner is **bit-exact (μ=384) to slightly conservative (μ=1200 +0.45 % over)** on non-bispinor at the production config. Both data points sit inside (and on the optimistic side of) bispinor's [−0.5 %, −10.8 %] window — the user's "10 %-ish" expectation easily holds, with no surprises in this regime.

The earlier 2026-05-18 sweep's pathological %-err numbers (Agent A +185 % at μ=192, −25 % over at μ ≥ 768; Agent C −13.9 % at 3×3×3 nb=200; Agent B −96.5 % at 2×2×2) were artifacts of the scope errors flagged in `SYNTHESIS.md`'s addendum:
- The 8 GB framework floor showed up large fractionally only because the algorithmic peak was small (small μ on a small system).
- The `cusolverMp + BFC@0.95` clash that pushed Agents A/C into measurement-protocol contortions was hbm40g vs hbm80g, not a sandbox bug.
- The "single-chunk over-prediction" hypothesis from Agent A (μ ≥ 768 over-predicts by ~25 %) is **falsified here** — at the same μ=384 single-chunk regime in production config, planner matches to 0.05 %. Whatever Agent A measured as "over-prediction" wasn't a planner issue.

The Agent C all_gather-slab hypothesis (−13.9 % at 3×3×3 nb=200) has **no analogue here** — the production config closes within 0.5 %. Worth a single quick check at 3×3×3 nb=200 production-config before declaring the hypothesis dead, but the bias direction has reversed, so the slab term, if it exists, isn't structurally biasing the production HWM.

The 8.04 GB at `zeta_fit_start` is the CUDA-context + cuFFT + NCCL constant floor — same magnitude observed in the earlier scope-erroneous runs. It is **not** the planner's job to model; the planner's HWM tracks the algorithmic peak above this floor. At μ=384 the algorithmic peak (20.13 − 8.04 = 12.09 GB) plus floor (8.04) is consistent within rounding with HWM_pred (20.12 GB) — the planner already correctly models the algorithmic contribution and absorbs the floor into its empirical-truth calibration via the safety margin.
