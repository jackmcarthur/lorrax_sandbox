# GN-PPM Sigma Performance Profile

**System**: MoS2 1x1 (Gamma-only, 80 bands, 640 centroids, nspinor=2)
**Platform**: Perlmutter A100-40GB
**Runs**: `runs/MoS2/00_mos2_1x1_profiling/`

## Current state (after `profile_ppm` branch)

The GN-PPM sigma stage had two serial bottlenecks. The first has been fixed; the second
has been identified, cached, and partially moved toward shipped precomputed assets.

| Bottleneck | Where | Before | After | Status |
|------------|-------|--------|-------|--------|
| XLA compilation of eager ops | tau-node loop in `_convolve_sigma_branch_kij` | ~410 s | ~2 s | **Fixed** via `@jax.jit` kernels |
| Crossing minimax solver (cold exact) | `_build_three_sigma_windows` -> `solve_phase_minimax_bandwidth` | ~95 s | ~95 s | **Identified** by xprof |
| Repeated minimax solves | `minimax_screening.py` | ~95 s | ~0 s lookup | **Fixed** via persistent disk cache |
| Shippable minimax tables | `src/common/minimax_assets/` | n/a | sub-ms lookup | **Started** for bundled reuse |
| Actual GPU sigma work | tau-loop einsums + FFTs | ~5 s | ~2 s | Healthy |

End-to-end on the profiling case:

| Metric | Original | After JIT + cleanup | With populated minimax disk cache |
|--------|----------|---------------------|----------------------------------|
| `compute_sigma_c_ppm_omega_grid` | 421 s | 105 s | 17.1 s |
| `gw_jax.ppm_sigma` (includes PPM build) | 417 s | 107 s | 18.7 s |
| Total wall time | 428 s | 137 s | 45.6 s |

## What was fixed: JIT kernels

The tau-node loop previously dispatched 5 eager JAX operations per iteration (einsum,
exp, iFFT, multiply+FFT, einsum). XLA compiled each independently on first call for
complex128 tensors of shape `(2,640,2,640,1,1,1)`, costing ~410s total.

The `profile_ppm` branch wraps the per-node channel build and omega accumulation in
`@jax.jit` kernels. XLA now compiles one fused program and reuses it across all ~260
tau-node iterations. The fix lives in `ppm_sigma.py:_convolve_sigma_branch_kij`.

Validated outputs (text-identical `eqp0.dat`, `sigma_freq_debug.dat`; byte-identical
`sigma_mnk.h5`) are preserved in:
- `01_lorrax_profile_ppm_jit/` -- first JIT pass
- `02_lorrax_profile_ppm_cleanup/` -- dead-code removal + module-level helpers

## What remains: crossing minimax solver

After the JIT fix, xprof traces show the remaining dominant cost:

| Call chain | Time (s) |
|------------|----------|
| `convolve_kij(omega >= E_F)` | 95.6 |
| ...`_build_three_sigma_windows` | 98.4 |
| ......`solve_phase_minimax_bandwidth` | 94.3 |
| .........`solve_crossing` -> `_polish` (Lawson/VarPro) | 89.7 |
| `convolve_kij(omega < E_F)` | 6.6 |

The positive-energy branch constructs a "core" window via a crossing minimax solve
(HGL quadrature, Lawson polish, VarPro LM). This solver runs on the host in pure
Python/NumPy and takes ~95s -- nearly all of the remaining PPM sigma time.

The solver code path is:
- `ppm_sigma.py` -> `_build_three_sigma_windows` -> `solve_phase_minimax_bandwidth`
- `minimax_screening.py` -> `solve_phase_minimax_bandwidth`
- `common/minimax.py` -> `solve_crossing` -> `_polish`, `_cr_minimax_lp`, `_cr_varpro_lm`

Fix strategies (not yet attempted):
1. Quantize `A_dim` upward and reuse bundled shipped crossing tables
2. Extend the shipped crossing sweep toward `A_dim = 200`
3. Reduce `crossing_max_nodes` / Lawson iteration count where accuracy allows
4. Cheaper closed-form rule for the core window parameters

## Minimax caching and shipped assets

Two new reuse layers now sit in `src/gw/minimax_screening.py`:

1. **Persistent exact cache**
   - Controlled by `LORRAX_MINIMAX_CACHE_DIR`
   - Disabled with `LORRAX_DISABLE_MINIMAX_DISK_CACHE=1`
   - Exact-key cache for repeated solves with the same `(range, target_error, eps_q, target_kind, max_nodes)`

2. **Shipped asset catalog**
   - Stored under `src/common/minimax_assets/`
   - Runtime reads `catalog.json` with `importlib.resources`
   - Controlled from GW input with `regenerate_minimax_tables = false`
   - Selection rule:
     - round requested range **up**
     - round requested error target **down** to a stricter or equal shipped table
     - reject tables whose node count exceeds the caller’s `max_nodes`

This keeps the current solver semantics intact while avoiding expensive host-side tuning.

### Error convention clarified

Reading `src/common/minimax.py` confirmed the current minimax solvers use **absolute
L-infinity error on the target function**, not an endpoint-relative criterion.

- Noncrossing tables are fit on `[1, R]` with
  `max_y |1/y - approx(y)| <= error_bound`
- When rescaled to a physical interval `[x_min, x_max]`, the absolute error scales as
  `error_bound / x_min`
- Crossing tables are fit on `[0, A_dim]` with
  `max_u |G(u) - approx(u)| <= error_bound`

### Current bundled tables

Generated by `tools/generate_minimax_assets.py`.

| Family | Sweep | Error bounds | Current status |
|--------|-------|--------------|----------------|
| Noncrossing | 13 `R` values from `1e1` to `1e5` (3 intervals/decade) | `1e-6`, `2e-7` | **Complete**: 26 tables |
| Crossing | `A_dim = 20, 40, 60, ..., 200` | `1e-6`, `2e-7` | **Seeded**: `20, 40` at both errors, `60` at `1e-6` |

The exact crossing sweep is intentionally left resumable because higher-`A_dim`,
stricter-error solves are still very expensive.

### Generator behavior

`tools/generate_minimax_assets.py` now:
- supports `--family crossing` / `--family noncrossing`
- writes a human-readable `README.md` plus machine-readable `catalog.json`
- preserves already-generated tables on `--no-clobber`
- merges with existing catalog entries so mixed-family resumptions stay clean

### Runtime validation

With the currently bundled tables:

| Request | Flag | Result | Time |
|---------|------|--------|------|
| `solve_laplace_minimax_interval(2, 2000, 1e-6)` | shipped `true` | bundled `R=1000`, `1e-6` table | `0.002369 s` |
| `solve_laplace_minimax_interval(2, 2000, 1e-6)` | shipped `false` | exact cached solve path | `0.000703 s` |
| `solve_phase_minimax_bandwidth(30, 1e-6, eps_q=1e-3)` | shipped `true` | bundled `A_dim=40`, `1e-6` table | `0.000513 s` |
| `solve_phase_minimax_bandwidth(30, 1e-6, eps_q=1e-3)` | shipped `false` | exact crossing solve path | `10.334154 s` |

So the shipped path is effectively free whenever a matching table exists, and for
crossing windows it is already orders of magnitude faster than solving nodes on demand.

### The evidence for XLA compilation as the original bottleneck

The first `_convolve_kij(pos)` took 412.7s; the second `_convolve_kij(neg)` took 7.0s.
Same functions, same shapes -- the difference is compilation cache. Device-gets were
instant (<0.1s). Multi-GPU (4x A100) gave zero speedup because compilation dominated.

## Other findings

### JAX distributed init double-call (fixed)

`python -m gw.gw_jax` runs module-level `_maybe_init_jax_distributed()` as `__main__`,
then `gw_init.py` re-imports it as `gw.gw_jax`, calling init again. Fixed with env-var
sentinel `_LORRAX_JAX_DISTRIBUTED_DONE`.

### Multi-process `device_get` crash (not fixed)

With `srun -n 4`, `minimax_screening.py:extract_gn_ppm_parameters_from_Wc` calls
`jax.device_get()` on arrays sharded across processes. Needs
`jax.experimental.multihost_utils.process_allgather`.
